# Gemini Claw - 基于 Gemini CLI 的智能体

Gemini CLI 本身已经是一个功能强大的全能智能体——它几乎可以做任何事情。
唯一缺失的重要部分是允许用户远程控制它（例如，通过 Discord）的通信渠道。

# 核心架构

消息流：
```text
入站渠道 (例如 Discord) -> SQLite 数据库 -> 轮询循环 (Python 异步任务) -> Gemini CLI 子进程 -> 出站响应
```

# 组成组件

**1. 渠道 (Discord 机器人)**
- **入站：** 使用 `discord.py` 监听 Discord 频道中的提及、白名单用户（如果不在对话线程中且没有明确提及他人），或特定的命令。解析用户的消息并将其插入到 SQLite 数据库中，状态设为 `pending`。
- **出站：** 一个与机器人并行运行的异步任务会持续轮询 SQLite 数据库，查找标记为 `completed` 或 `failed` 的消息。一旦找到，它会将存储的响应发送回对应的 Discord 频道（或通过响应中的 `[to_channel: <channel_id>]` 标签指定的频道），并将记录更新为 `delivered`。

**2. 数据库层**
- 使用 SQLite 作为解耦层，以确保当执行耗时较长或机器人崩溃时消息的持久性与可靠性。
- **表 `messages`：**
  - `id` (INTEGER PRIMARY KEY AUTOINCREMENT)
  - `channel_id` (TEXT)
  - `message_id` (TEXT)
  - `prompt` (TEXT)
  - `status` (TEXT) - 状态可选为 `pending`, `processing`, `completed`, `failed`, `delivered`
  - `response` (TEXT)
  - `created_at` (TIMESTAMP)

**3. 轮询循环与子进程执行**
- 一个与 Discord 机器人并发运行的 Python `asyncio` 循环。
- 它轮询数据库中的 `pending` 消息（过滤掉正处于忙碌状态的频道）并将其标记为 `processing`。
- 它通过生成后台 `asyncio` 任务并发地处理消息，同时确保来自同一个对话线程的消息仍然按顺序串行处理。
- 它使用无头的 Python 子进程来执行 Gemini CLI。
  - **命令执行：** 为避免通过 Discord 转发 PTY 交互的复杂性，我们结合 Gemini CLI 的无头模式 (`-p`) 与 YOLO 模式 (`-y` 或 `--yolo`) 来自动接受工具。在 `config.toml` 的 `[gemini]` 配置段中定义的任何 `policy` 文件，都将自动作为 `--policy <file>` 参数追加。
  - **执行命令格式：** `gemini -y -p "<prompt>"`（我们还可能添加 `-r latest` 以恢复状态，或手动管理每用户/频道的会话）。
- 捕获标准输出和标准误差。一旦进程完成，输出将保存至 `response` 字段，状态更新为 `completed`（如果退出码非零则设为 `failed`）。

**4. 初始化与配置命令**
- 为了在首次启动期间引导用户，我们提供了一个命令行界面：
  - `uv run geminiclaw init`：当用户首次克隆存储库时运行。它将所有资源从内置工作区复制到当前目录（跳过现有文件，若内容不同则打印警告），并初始化 SQLite 数据库模式。
  - 命令行既处理启动机器人 (`uv run geminiclaw start`)，也管理针对 macOS 和 Linux 的后台服务 (`uv run geminiclaw service`)。
- 这些命令作为入口点生成了 `src/geminiclaw/cli.py` 中对应的 Python 实用功能，使核心工作全部在 Python 端处理。
 
**5. 附件处理**
- **入站 (下载)：** 任何附加在提及机器人的消息中的附件都会被下载到一个可配置的目录下（例如，工作区内部）。
- **提示词增强：** 机器人会在提示词末尾附上一个 `Attachments:` 列表，引用已下载的文件（例如：`- attachments/message_id_filename.ext`）。这会告知智能体文件的可用性。
- **智能体访问：** 由于文件存放在工作区或经 `--include-directories` 包含在内，Gemini CLI 智能体可以使用其自有工具（如读取文件）来访问它们。
- **出站 (传输)：** 智能体可以通过在响应中包含语法 `[attachment: path/to/file]` 将文件发送回用户。机器人从最终响应中提取这些路径，验证文件是否存在于工作区内，并在后续消息中安全地将它们作为 Discord 附件上传发送。
 
**6. 定时任务 (Cronjobs) 管理**
- **触发：** 使用 `apscheduler` 基于 `cron` 计划表达式周期性触发。同时支持一个可选的 `probability` 浮点数参数，实现按给定概率随机跳过执行。
- **流程：** 读取提示词文件，并在 SQLite 数据库中插入一条状态为 `pending` 的消息，记录目标频道的 ID 并将 `author_id` 设为机器人自身的 ID。线程的创建被推迟到成功收到输出响应后（避免遇到 `NO_REPLY` 产生空线程）。定时任务执行始终关闭流式传输。
- **处理：** 标准的异步轮询循环会自动抓取该任务，交由 Gemini 智能体执行，并将响应传送回相应的线程中，与普通用户提示词无异。针对静默消息，执行完全在后台进行，没有任何的 Discord 交互。这使得调度与执行逻辑彻底解耦。
 
# 进程管理
- **单进程：** Discord 机器人（`discord.py` 客户端循环）和数据库轮询机制通过 `asyncio` 任务运行在同一个 Python 进程内。这避免了管理多个后台服务的开销和复杂性，同时仍能依托 SQLite 层的解耦保证架构的独立性。
- **结束进程：** 机器人在字典中记录着运行中的 `gemini` 子进程。当频道中接收到 `-kill` 命令时，会针对该频道（或对话线程）手动终止进程。
- **重启进程：** 接收到 `-restart` 命令时，机器人会生成一条后台 shell 命令 `geminiclaw service restart` 以重新启动机器人服务本身。请谨慎使用，因为这会导致机器人暂时断开连接。
- **终止活动命令：** 如果机器人正在运行耗时较长的提示词而你想终止它，请在线程中发送 `-kill`。这会强行杀死当前线程正在运行的 `gemini` CLI 进程。

# 多机器人聊天

Gemini Claw 原生支持多个机器人的互动！你可以运行机器人的多个实例（配置不同参数或系统提示词），让它们互相交流，或在同一个线程中与多位用户对话。

- **无缝参与对话线程：** 在一条消息中提及多个机器人（例如：`@Bot1 @Bot2 让我们讨论 Python`）。它们会自动协同并进驻同一线程，而不会产生任何重复或冲突。
- **智能流式处理：** 机器人在生成 token 时会向消息末尾追加 `(incomplete)` 标志。其他监听的机器人会等待一条消息完全生成完毕后才做出答复，防止它们在句中发生互相打断的情况。
- **暂停对话：** 如果机器人陷入了无休止的相互交谈中，你可以在线程内发送 `-stop` 命令。这会将该线程对于所有监听的机器人都标记为非活跃状态，从而停止自动回复循环。
- **恢复对话：** 如果你希望机器人们重新开始倾听该线程，只需输入 `-continue`。机器人们将再次激活并恢复参与交流。
 
# 附件支持
 
Gemini Claw 支持双向的附件处理功能：
- **入站：** 自动下载 Discord 消息的附件（文件、图片等），并使其在工作区内对 Gemini CLI 智能体可用。
- **出站：** 智能体可以通过在其响应中包含语法 `[attachment: path/to/file]`，将工作区中的文件发送回给你。机器人会自动提取并将这些文件上传至 Discord。

# 频道路由

智能体可以通过在响应中加入 `[to_channel: <channel_id>]` 语法，将其响应直接路由到特定的 Discord 频道而不是当前频道。机器人会提取该频道 ID，抓取对应频道并将响应原生送达那里。该标签将从最终的消息内容中被自动移除。
